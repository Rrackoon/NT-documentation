# Стратегия хранения данных

## Подход

Для NutriTrack выбран `polyglot persistence`: разные типы данных хранятся в тех системах, которые лучше всего подходят под их профиль нагрузки

## Итоговый выбор

| Сущность | Тип хранилища | Инструмент |
| --- | --- | --- |
| User | Реляционная СУБД | PostgreSQL |
| Meal | Реляционная СУБД с партицированием | PostgreSQL |
| MealItem | Реляционная СУБД с партицированием | PostgreSQL |
| Product | Реляционная СУБД + поисковый индекс | PostgreSQL + Elasticsearch |
| WeightRecord | Time-Series DB | TimescaleDB |
| DailyGoal | Реляционная СУБД | PostgreSQL |

## Почему именно так

### PostgreSQL

- строгая схема и внешние ключи
- атомарность операций при создании `Meal` и `MealItem`
- низкий порог входа для команды

### Elasticsearch

- полнотекстовый поиск по названию продукта
- быстрые фильтры и ранжирование результатов

### TimescaleDB

- оптимизация под данные, привязанные ко времени
- удобные диапазонные запросы и агрегации
- естественный сценарий для построения графиков веса

### Redis

- кэш дневной статистики
- снижение нагрузки на backend и базу при частых просмотрах домашнего экрана

## Дополнительные замечания

- Для `Meal` и `MealItem` рекомендуется партицирование по дате
- Для `Product` нужен отдельный процесс синхронизации между PostgreSQL и Elasticsearch
- Для пользовательских блюд требуется `user_id` и флаг `is_custom`
